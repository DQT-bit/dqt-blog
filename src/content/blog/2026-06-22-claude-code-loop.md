先把用法甩出来：

```
/loop 你先把这些做了，好了告诉我
```

就这一行。我用这一句，把手头攒的几件小事一股脑扔给它，然后去倒了杯水。回来的时候，活儿全干完了，它才喊我。中间一次没来问我。

---

那天我手头攒了三件小事：

一篇文章要按之前的意见改几个地方；自己搭的小网站有个链接打不开要修；还有一堆待办事项的状态要更新一下。

都是那种不难、但一件件做很烦的杂活。

平时我得盯着它一件件做：改文章要来回确认好几次，修网站要等它改完、上传、再等网站更新、再点开看好没好，更新待办还得一条条核对。每一步我都得在旁边看着。

这次我没盯着。撂下一句"你先把这些做了，好了告诉我"，我就去忙别的了。

几分钟后回来，三件事全办妥了，它一条条告诉我哪件做完了、验过没问题。修网站那件最有意思——它改完之后那个网页不会立刻刷新，得等一两分钟。这种要干等的场合，它平时多半会停下来问我一句"要不要等"。这次它没问，自己就等下去了，一遍遍去看，直到网页真的刷新出来，才算这件过。

---

`/loop` 这个用法，我以为大家都知道，结果问了一圈——用 Claude Code 大半年的人，都没用过。

所以写一篇说说。

---

它干的事很简单：把一句话交给它，让它自己反复做，做到它觉得够了为止。

有两种用法。

**第一种，定个时间，每隔一会儿做一次：**

```
/loop 5m 看一下网站发布好了没
```

这里的 `5m` 就是五分钟。它每五分钟去看一眼，到点自己醒过来。适合那种"盯着一件事的进展"——比如等一个东西上线、等一个长任务跑完、看看有没有新消息。

**第二种，不定时间，让它自己看着办：**

```
/loop 把这几件事做完，做完告诉我
```

它自己判断要不要接着干。一段做完觉得行了就收工；要是还在等什么东西（比如等网站更新），就自己定个时间再回来看。

十次里有九次，用第二种就够了。

---

跑歪了怎么办？

直接关掉，或者打断它，重新说话就行。它不是个一旦启动就停不下来的东西，你随时能接管。

---

说几个我和身边人真拿它干过的活，你大概能对上号。

**第一类，攒一堆小事打包扔过去。**

我那次的三件事就是。再比如：把一个文件夹里几十张图片全压缩一遍；把一份表格里的日期格式统一改一下；把十几个笔记文件的标题按规则重命名。

这种活的特点是——每件都简单，但一件件做，人会烦。攒成一句"这几样弄完叫我"，扔了就走。

**第二类，盯着一件慢慢出结果的事。**

等一个大文件上传完、等一段程序跑完、等一个网页改完审核通过。这些事你干等着也是等，盯屏幕还累。交给它："好了告诉我"，你该干嘛干嘛去。

我修网站那次就是这样——它改完之后那个网页不会立刻更新，得等一两分钟。它就自己等着，一遍遍去看，看到真的好了才喊我。

**第三类，等一个外面的消息。**

等同事回你一条评论、等某个页面出现新内容、等一个数字变化。它替你盯着，来了第一时间告诉你。

---

但有些活别交给它：

你想一步步看着它做的——比如写文章，你想看着它一段段写、随时插话，这种就别用，它一口气写完反而不是你要的。

涉及花钱、对外发东西、删东西这种——得你亲自点头，别让它自动办了。

还有那种没有明确"算做完"标准的。比如你说"研究一下某个话题"，它根本不知道做到哪儿算研究完，可能就一直转。

最后这条最关键：交给它之前，你心里得有个清楚的"做完长啥样"。话说得越具体——"把这三件弄完告诉我"，它越知道什么时候该收工。含糊地说一句"帮我搞搞"，它也只能含糊地还你。

---

还有个东西容易混，顺便说清楚。

`/loop` 只在你这次开着的窗口里跑，窗口一关就停了。如果你要的是"每天早上八点给我发条提醒"这种长期的、关了电脑也得跑的，那是另一个命令 `/schedule` 管的事，它跑在云端，不关窗口的影响。

简单记：临时盯一会儿用 `/loop`，长期定时用 `/schedule`。隔得太久（比如超过一个钟头才做一次）的，也直接用 `/schedule`。

---

第二种用法还有个细节挺聪明。

它会自己掂量该隔多久看一次：手头能马上干完的，干完就收工；在等外面一件事的，盯紧一点；啥动静都没有的，就把间隔拉长——每检查一次都得耗一点额度，没事就别白耗。

它自己定的间隔，短则一分钟，长到一个钟头封顶。

定时间隔像个闹钟，到点就响；这种自适应的，更像个有分寸的同事，知道什么时候该催、什么时候该等。

---

上手就一句话：

```
/loop 你要它做的事
```

试试看。

---

*我最近用它干的几件事：等一个自动化流程跑完、反复打磨一个项目的说明文档、陪一个小工具从头到尾走一遍。凡是"你先做，做完叫我"的活，都能这么使。*
